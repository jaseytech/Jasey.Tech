Salesforce Interview Questions

Why are Visualforce pages served from a different domain?

Visualforce pages are served from a different domain to improve security standards and block cross site scripting.

What is the difference between a Role and Profile in Salesforce?

A profile will ultimately control access to which records a user has in a Salesforce org. No user can work on the Salesforce org without being assigned a profile. The Profile is therefore mandatory for every user.

Role however is not mandatory for every user. The primary function of the Role/ Role hierarchy is that it allows higher level users in hierarchy get access to records owned by lower level users in the hierarchy. An example of that is Sales Managers getting access to records owned by Sales Reps while their peers do not get access to it.

Tell me about Data Skew in Salesforce?

“Data skew” is a condition which you will encounter when working for a big client where there are over 10,000 records. When one single user owns that many records we call that condition ‘ownership data skew’.

When such users perform updates, performance issues will be encountered because of “data skew”. This happens when a single user/ members of a single role own most of the records for a particular object.

What is the use of writing sharing rules? Can you use sharing rules to restrict data access?

Sharing rules are written to give edit access (public read and write) or public read only access to certain individuals in Salesforce org. A classic example is when:- only your managers or superiors need to be given extra credentials to your records in objects as compared to your peers.

By default, all users in your organization will have organization-wide-default sharing settings of either Public Read Only or Private.
To give access to more records, which users do not own, we write sharing rules.
Example: Sharing rules are used to extend sharing access to users in public groups or roles. Hence, sharing rules are not as strict as organization-wide default settings. They allow greater access for those users.

As far as the second part of the question is concerned, the answer is no. We cannot use sharing rules to restrict data access. It is only used for allowing greater access to records.

What are the different types of object relations in salesforce? How can you create them?

The different types of object relationships in Salesforce are:

  1. Master-Detail Relationship (1:n):- It is a parent-child relationship in which the master object controls the behavior of the dependent child object. It is a 1:n relationship, in which there can be only one parent, but many children.The main concept you need to be know is that, being the controlling object, the master field cannot be empty. If a record/ field in master object is deleted, the corresponding fields in the dependent object are also deleted. This is called a cascade delete. Dependent fields will inherit the owner, sharing and security settings from its master.You can define master-detail relationships between two custom objects, or between a custom object and standard object as long as the standard object is the master in the relationship.
  2. Lookup Relationship (1:n):-
    Lookup relationships are used when you want to create a link between two objects, but without the dependency on the parent object. Similar to Master-Detail relationship, you can think of this as a form of parent-child relationship where there is only one parent, but many children i.e. 1:n relationship.The difference here is that despite being controlling field, deleting a record will not result in automatic deletion of the lookup field in the child object. Thus the records in the child object will not be affected and there is no cascade delete here. Neither will the child fields inherit the owner, sharing or security settings of its parent.
  3. Junction Relationship (Many-To-Many):-
    This kind of a relationship can exist when there is a need to create two master-detail relationships. Two master-detail relationships can be created by linking 3 custom objects. Here, two objects will be master objects and the third object will be dependent on both the objects. In simpler words, it will be a child object for both the master objects.
Describe considerations when developing in a multi-tenant environment

No client software is required for access and development.

Using an IDE (like Eclipse) is optional

Auto updates are applied three times a year and cannot be scheduled.

Auto-updates are applied to customers at the same time.

Sandboxes updated prior to production environments, so changes can be pre-viewed and tested.

All customers share the same code base and database in a Salesforce environment

All customizations are specified as metadata for easy upgrades

All resources are shared (CPU, etc)

Code cannot be deployed into production unless test coverage is achieved

Common use cases for declarative customization?

Display sum or count of the number of related records (use rollup summary fields)

Display average of number of related recs (use rollup summary fields together with formula field)

Validate that country code is valid ISO code (validation rule)

Ensure that a reason is entered before closing an opportunity (validation rule)

Ensure that value is in a valid format (Regex function in validation rule)

Describe the capabilities and use cases for formula fields

Formula fields are read-only fields that auto-calculate a value based on other fields or formulas.

These are auto-updated when one of the fields referred to in the formula changes.

Can be based on simple formula or advanced formula

Simple formulas allow selection of one or more fields and operators such as Add, Subtract, or Concatenate.

Advanced formulas allow the selection of one or more fields, operators, and advanced formulas.

Merge fields can reference fields in related objects, such as custom formula field on contact can reference a field from account.

Can be of type:

  • checkbox
  • currency
  • date
  • date/time
  • number
  • percent
  • text

Use cases:

  • Concatenate fields
  • Obtain value of field from parent record on child record
  • obtain value of a field from a related object (lookups) up to 10 relationships away (cross object formula)
  • Perform date calculations (like difference between dates, finding month from date, adding days to date)
  • Display images based on certain values
  • Use conditional logic to calc values
  • Determine value from a set of values
  • Create URL link based on field value
  • Calculate commissions based on the value of other fields
  • Remove formatting from a field value
  • Create formatted addresses including spaces and line breaks based on country value
  • Perform conversions

Operators used in Formulas:

  • MATH – Add, Subtract, Multiply, Divide, Exponent, Parenthesis to specify order of evaluation)
  • Logical – Equal, not equal, less than, etc
  • Text & concatenate

Functions used in Formulas:

  • Date & Time – Now, Today, Day, Month, Year, Date, DateValue, DateTimeValue)
  • Logical – AND, CASE, IF, IsChanged, isNew, isNumber
  • Informational – BlankValue, isBlank, isNull, PriorValue
  • Advanced – Regex, Vlookup
  • MATH – ABS, Ceiling, Distance, Exp, Floor, Geolocation, LN, Log, Max, Min, Mod, Round, SQRT
  • Text – Begins, Br, CaseSafeId, Contains, Find, getSessionID, HyperLink, Image, Includes, IsPickValue, Left, Len, Lower, LPad, Mid, Right, RPad, Substitute, Text, Trim, Upper, Value
Describe Capabilities of declarative processes automation features

Workflow – Used to update fields (record or parent), create task, send email, send outbound message

Process Builder – Create record, update fields in any related record, submit for approval, send email alert, call visual workflow, post to chatter, call Apex, Quick Action, invoke a process

Visual Workflow – Accept user input, call Apex, create records, delete records, post to chatter, send email, submit for approval, date fields in any rec, Quick Action, query records, loop records, multiple decisions

Approval Process – Update fields (record or parent), create task, send email alert, send outbound message

Describe the capabilities of Campaigns

Campaign – a way to manager your marketing efforts

Capabilities of Campaign object include:

  • Campaign planning – record the type/dates/budget/costs/expected revenue of Campaign
  • Campaign layouts -customizable just like any other object in Salesforce (create custom fields, record types, layouts, validation rules)
  • Campaign Management – record stats, budgeted cost, actual cost
  • Campaign Hierarchy – relate campaigns up to 5 levels
  • Track Campaign stats – track number of responses, leads, opportunities, and opportunity value generated
  • Campaign Influence – two types: original and customizable
    • Original allows campaigns to be associated to opportunities automatically
      • Can set time frame and filter criterial for association
    • Customizable allows you to set up revenue attribution / opportunity across man different campaigns
      • can even integrate with third parties to send email blasts using a combination of Templates and Campaigns

Campaign Members – Junction object used to create many-to-many relationships between contacts/leads and Campaigns

  • Capabilities of Campaign Member object include:
    • Select Leads of Contacts to associate with Campaign
    • Several options for how to add members to a Campaign
    • Track responses – update member status manually or via import file using a tool like CRMFusion
Describe the capabilities of Lead Object

Capabilities of the Lead object include:

  • Social Leads – link accounts to social network profiles and view activity/feeds
  • Define Lead Business Process – Lead status values and associate with record types
  • Define a path tailored to your company’s Sales process
  • Web-To-Lead – Generate HTML to add to a website to capture Leads into Salesforce
  • Web-to-Lead Auto Response Rules – send an auto reply back after filling in a Web-to-Lead form, including logic/criteria to determine which email template and content to use
  • Import Leads from a CSV File
  • Lead Management – Source/owner User or Queue/status
  • Lead Queues – holding location for Leads that have not yet been assigned an owner
  • Lead Assignment Rules – automatically assign leads to users or queues based on field criteria
  • Lead Conversion – create or select Account and Contact. optionally create Opportunity
Describe capabilities of Account objects

Capabilities of Account object include:

  • Business Accounts – for Business to Business
    • Business Accounts can be merged with other Business Accounts
  • Person Accounts – for Business to consumer
    • not enabled by default
    • combo of standard account and contact fields and 1 account and 1 contact record
    • must be enabled by Salesforce Support and cannot be disabled
    • Person Accounts can be merged with other Person Accounts
  • Social Accounts – link accounts to social networking profiles and view activity/feeds
  • Account Hierarchy – record parent/child relationship between companies
    • Salesforce Classic will show up to 200 levels while Lightning shows up to 2,000 accounts
  • Account Teams – automatically or manually assign users that work together on account
    • role and access to account and related opportunities and cases
  • Transfer one or multiple accounts to new owners and check a box to quickly email a new owner to notify them
  • Contact Roles – track roles each contact plays in Account and who is the primary contact
  • Merge Accounts – remove duplicate account records
    • recently available in Lightning, but done differently than in Classic
Describe the capabilities of Contact Object in Salesforce

Capabilities of the Contact Object include:

  • Import Contacts
  • Social Contacts – link contacts to social networking profiles and view activities/feeds
  • Org chart – use ‘Reports to’ field to define reporting relationships and generate an org chart
  • Send Emails and track responses
  • Merge Contacts – remove duplicate contact records
    • Recently available in Lighting
  • Contacts can be linked to multiple accounts

Contacts to Multiple Accounts:

  • To enable, you must allow users to relate to multiple Accounts and add the related Contacts related list to Account page layout
    • You can then remove Standard Contacts related
    • Must add Related Accounts related list to Contact page Layout
  • One account can be linked to multiple accounts
  • Once an Account is selected as a primary Account and is a direct relationship, the primary account will be listed on the Contact record in Account field
  • You can choose if Contact activities roll up to the primary account
  • Other accounts are secondary/indirect relationships
  • Deletion of an account can be blocked if it has direct contacts that are related to other accounts
  • Deletion of Account can be allowed even if it has direct contacts that are related to other accounts
    • Direct contacts will also be deleted
  • If primary accounts on contact record is replaced, either relationship can be changed to indirect or the relationship can be deleted
  • This can also be used for indirectly relating Person accounts to Business accounts
    • need to add the Related Accounts related list to Person Account page layouts
  • You’ll need to create custom report types for this as they don’t come standard with Salesforce
Describe the capabilities of Activities object

Capabilities of Activities include:

  • Includes Tasks and Events
  • Activity History is a related list of completed tasks and expired events
  • Synchronize with Outlook
  • Lightning and Outlook
    • Available in both Classic and Lightning
    • Free
    • Cloud-based, no downloads or updates on end user end
  • Paid option: Salesforce Inbox
  • Link to related records such as Accounts, Contacts, and Opportunities
